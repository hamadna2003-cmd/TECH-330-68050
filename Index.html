PROTOTYPE DESIGN PROCESS
▶ 1.5.1  3.1 Interface Planning
When we started designing the interface, the first thing we did was think about who would actually use this prototype. We identified three main groups: students who are learning about VR/AR systems, teachers who might use it in their classes, and technical folks who want to understand how everything works under the hood. Each group has different needs, and we wanted to make sure the interface worked well for all of them. We took a user-centered approach, which means we created detailed profiles of typical users (called personas) and kept them in mind throughout the design process. These personas helped us make decisions about everything from how we organize information to what words we use. For instance, realizing that not everyone has a strong technical background pushed us toward using clear, descriptive labels and straightforward navigation instead of throwing around technical jargon or creating complicated menu systems. We organized the content into four main sections that made logical sense. The Project Objective section tells you what this prototype is all about and what you’ll learn from using it. The System Block Diagram section is for those who want to see the technical architecture visually laid out. The Prototype Design section gets a bit meta—it explains how we actually designed and built the thing you’re using. And finally, the Exit option gives you a clear way out, because nobody likes getting trapped in an application with no obvious way to leave. For the visual design, we wanted something that looked professional and polished but wasn’t intimidating. We chose colors that have good contrast for readability (meeting accessibility standards), selected fonts that work well on any device, and established consistent spacing to keep everything looking organized. We also made sure buttons and other interactive elements clearly show what state they’re in—whether you’re hovering over them, clicking them, or if they’re disabled. Rather than adding lots of decorative flourishes, we kept things clean and focused on making the interface easy to understand and use. We also had to plan for the fact that people might use this on anything from a big desktop monitor to a small phone screen. The layout automatically adjusts—on wide screens, navigation might appear as a horizontal bar, while on phones it collapses into a hamburger menu. We made sure buttons are big enough to tap easily on touchscreens (at least 44x44 pixels is the sweet spot), but we didn’t waste space on desktop layouts where precision clicking is easy.
▶ 1.5.2  3.2 User Flow Design
User flow design is basically mapping out how people move through the prototype—where they can go, what choices they need to make, and how the system responds. This helped our team get on the same page about how the prototype should behave and prevented confusion during development. Here’s how a typical session goes: You type in the URL and the main menu appears. This menu is your home base—everything branches out from here. You see four clearly labeled buttons for the main sections. When you hover your mouse over a button (or tap it on a touchscreen), it reacts visually to let you know it’s clickable. This little bit of feedback is important because it confirms you’re about to do something before you fully commit. When users select the Project Objective option, the system transitions to a dedicated page presenting the prototype’s purpose, learning objectives, and contextual information about VR/AR technology applications. This page includes text content organized with clear headings, supporting imagery or diagrams, and a navigation element (typically a “Back” button or persistent navigation menu) that allows return to the main menu. The flow supports both linear reading and random access, accommodating different user preferences and interaction styles. Selecting the System Block Diagram option triggers transition to a page displaying the architectural diagram with accompanying explanatory text. This page implements zoom and pan functionality (where technically feasible) to allow detailed examination of diagram elements. Hover interactions or tap targets on diagram components reveal tooltips or modal dialogs with additional information, supporting progressive disclosure of complexity. Users can explore the diagram at their own pace before returning to the main menu. The Prototype Design option navigates to documentation of the design process, including user flow diagrams, interface mockups, and rationale for design decisions. This self-referential content provides transparency about development methodology and supports educational objectives around design thinking and iterative development processes. The page structure mirrors other content pages, maintaining consistency and reducing cognitive load. The Exit option presents a confirmation dialog asking users to verify their intent to leave, preventing accidental navigation away from the application. Upon confirmation, the system may redirect to an external page, close the browser tab (where browser security policies permit), or display a farewell message. This pattern follows established conventions for application exit flows, meeting user expectations shaped by experience with other software systems. Error flows handle scenarios such as broken navigation links, missing content resources, or browser compatibility issues. The system implements graceful degradation, displaying informative error messages that explain the problem and suggest remedial actions rather than presenting cryptic technical errors or failing silently. Error states maintain the overall interface structure and navigation options, preventing users from becoming trapped in dead-end error conditions.
▶ 1.5.3  3.3 Navigation Logic
Behind the scenes, navigation logic is what makes moving between sections smooth and predictable. We built this as a single-page application, which is a fancy way of saying that when you navigate to a new section, the browser doesn’t reload the entire page—it just swaps out the content. This makes transitions feel snappier and more responsive. The system keeps track of where you’ve been, so your browser’s back and forward buttons work just like they do on any other website. This was important to us because people are used to those buttons working, and we didn’t want to break that expectation. We also update the URL in your browser’s address bar as you navigate, so each page has its own unique address that you can bookmark or share with others. State management employs a centralized store (implemented through vanilla JavaScript or a lightweight state management library) that maintains the current active page, user preferences, and any dynamic content. Navigation actions dispatch state updates that trigger rendering of appropriate content and updating of active navigation indicators. This architecture separates navigation logic from presentation concerns, facilitating testing and reducing coupling between components. Menu highlighting provides persistent visual indication of the current page, helping users maintain orientation within the application structure. The active menu item displays with distinct styling (different background color, underline, or icon) that clearly differentiates it from inactive options. This wayfinding support is particularly important in applications with complex navigation hierarchies, though the flat structure of this prototype reduces navigation complexity. Deep linking support enables users to access specific content pages directly through URL entry or external links, rather than requiring navigation from the main menu. The system parses URL parameters on initialization, determines the requested content, and renders the appropriate page while maintaining consistent navigation structures. This capability supports sharing of specific content sections and integration with external learning management systems. Accessibility considerations in navigation logic include keyboard navigation support (tab order following logical reading sequence, Enter/Space activation of controls), skip navigation links for screen reader users to bypass repetitive navigation elements, and ARIA landmarks that identify navigation regions for assistive technologies. Focus management ensures that keyboard focus moves logically during page transitions, preventing disorientation for non-visual navigation.
▶ 1.5.4  3.4 VR/AR Interaction Methods
While this prototype works like a regular website right now, we designed it with VR/AR interaction methods in mind. Understanding how people interact in immersive environments is crucial if we ever want to expand this into a full VR/AR experience. In VR, interaction is all about taking advantage of the 3D space around you. The most natural method is direct manipulation—just reach out with your virtual hand or controller and grab things, just like you would in real life. For objects that are far away, you can point at them with a laser-pointer-style beam from your controller. There’s also gaze-based interaction, where you select things just by looking at them for a moment—this is great for hands-free control when you’re holding something or your hands are busy. One of the trickier challenges in VR is moving around in large virtual spaces when you’re physically standing in a small room. That’s where teleportation comes in—you point to where you want to go, and boom, you’re there instantly. It might break immersion a bit, but it prevents that queasy feeling you get from smoothly gliding through virtual space while your body is standing still. Most VR apps use this method now because it’s a good compromise between comfort and practicality. AR interaction is different because you’re dealing with both the real and virtual worlds at once. On smartphones and tablets, you can just touch virtual objects on your screen with your finger—simple and intuitive. With headsets like HoloLens that let you see through them, you make gestures in the air that cameras detect. Voice commands are really useful in AR too, especially when using your hands would block your view or just be awkward—imagine trying to tap a menu while you’re also holding a real tool or examining a physical object. Object manipulation in VR/AR contexts extends beyond simple selection to include rotation, scaling, and physics-based interactions. Grab-and-throw mechanics leverage intuitive real-world schemas, where velocity at release determines projectile trajectories. Two-handed manipulation of large objects provides natural affordances for rotation and scaling through relative motion of hands. Snapping behaviors align objects to predetermined positions or orientations, reducing precision requirements and accelerating common tasks like assembly or arrangement. Menu and UI interactions in VR/AR contexts often employ diegetic interfaces integrated into the virtual environment rather than screen-space overlays. For example, menus might appear as virtual tablets or holographic panels that users manipulate like physical objects. This approach maintains immersion and leverages spatial memory, as users remember where interface elements exist in virtual space. Alternatively, head-up display (HUD) elements provide persistent information without occupying hand attention, useful for status indicators and notifications.
▶ 1.5.5  3.5 Accessibility and Usability Considerations
Accessibility isn’t something we tacked on at the end—it was part of our thinking from day one. The goal was to make sure everyone can use this prototype effectively, whether they have a disability or not. This is what universal design is all about: building things that work for everyone from the start. For people with vision problems, we made sure there’s enough contrast between text and backgrounds (at least 4.5:1 for regular text, 3:1 for larger text—that actually exceeds the WCAG 2.1 Level AA standards). We never rely on color alone to convey information; we always pair it with text, icons, or patterns so people with color blindness don’t miss anything. The text sizes are flexible too—they respect whatever size you’ve set in your browser, and the interface still works properly even if you zoom in to 200%. For users who rely on screen readers, we made sure everything is properly labeled and structured. Every image has descriptive text (called alt text) that explains what it shows, so someone who can’t see the image still gets the information. We used proper HTML5 structure tags that help screen readers understand how the page is organized. Where needed, we added ARIA attributes—these are special labels that tell assistive technology what different interface elements do and how they’re related. You can navigate the entire prototype with just a keyboard—no mouse required. This helps people with motor disabilities, but it’s also great for keyboard power users who find it faster than clicking around. You can Tab through all the interactive elements in a logical order, and press Enter or Space to activate them, just like you’d expect. We made sure there’s always a clear visual indicator showing which element has focus, so you never lose track of where you are. We even included some keyboard shortcuts for common actions, being careful not to use keys that would interfere with screen readers. Motor accessibility considerations include generous hit targets exceeding minimum touch target sizes, reducing precision requirements for users with motor control limitations. The interface avoids interactions requiring fine motor control such as click-and-drag or hover-dependent content revelation. Timeouts for interactive elements provide sufficient duration for users who interact more slowly, with options to extend or disable time limits where appropriate. The system avoids interactions requiring simultaneous multi-button presses or complex gesture sequences. Cognitive accessibility addresses users with learning disabilities, attention deficits, or cognitive processing differences. The interface employs consistent layouts and interaction patterns across all pages, reducing memory demands by establishing predictable structures. Clear, concise language avoids unnecessary jargon, with technical terms defined upon first use. Visual hierarchy through heading structures, whitespace, and typography guides attention to important information. Error messages provide specific, constructive guidance for remediation rather than generic failure notifications. Content organization follows logical groupings, with related information positioned proximally to reduce cognitive load. Usability testing throughout development involved representative users performing realistic tasks while observers documented difficulties, confusion, or unexpected behaviors. This empirical feedback guided iterative refinements, addressing issues such as unclear labeling, unintuitive navigation paths, or inadequate feedback for system states. Testing included users with diverse abilities and technical backgrounds to ensure the prototype serves its entire intended audience effectively.
▶ 1.5.6  3.6 Real-World Application Mapping
We didn’t design this prototype in a vacuum—we kept real-world VR/AR applications in mind throughout the process. By connecting our design to actual use cases in education, training, tourism, and other fields, we made sure the prototype stays grounded in reality rather than just theoretical concepts. Education is one of the most exciting areas for VR/AR. Imagine medical students being able to explore the human heart in 3D, taking it apart and putting it back together to really understand how it works. History students could walk through ancient Rome as it looked 2,000 years ago, experiencing the past in a way no textbook can provide. Science students might shrink down to molecular scale to watch chemical reactions happen, or zoom out to see how galaxies form. Our prototype’s navigation design follows these educational software principles—letting users explore at their own pace and gradually revealing more complex information as they go. Training is another area where VR/AR really shines. Surgeons can practice complex operations on virtual patients over and over until they’ve mastered the technique—no real patients at risk. Pilots train for emergency situations that would be dangerous or impossible to recreate in real life. Factory workers learn how to maintain equipment using AR glasses that overlay step-by-step instructions right onto the machine they’re working on. Our prototype’s clear, structured approach to navigation and instructions reflects how training software needs to be crystal clear and easy to follow, especially when you’re learning a critical skill. Tourism and cultural heritage applications use VR for virtual destination experiences and AR for on-site information enhancement. Prospective travelers explore hotels and attractions virtually, making informed booking decisions based on experiential preview rather than static photographs. Museum visitors point smartphones at exhibits to reveal contextual information, animations, or reconstructions showing artifacts in original contexts. Historical sites overlay AR representations of destroyed or modified structures, helping visitors visualize locations as they appeared in different eras. The prototype’s content presentation strategies—combining visual elements with descriptive text—mirror approaches used in cultural heritage applications. Industrial design and architecture employ VR/AR for visualization and collaboration. Architects present building designs as walkable virtual spaces, enabling clients to experience scale and spatial relationships that drawings cannot convey. Product designers evaluate form and ergonomics by interacting with virtual prototypes before manufacturing expensive physical models. Remote teams collaborate in shared virtual spaces, manipulating 3D models synchronously regardless of geographic distribution. The prototype’s clean visual design and emphasis on spatial information representation reflect practices in professional visualization applications. Healthcare applications beyond training include pain management through immersive distraction, exposure therapy for phobias and PTSD, and surgical planning through patient-specific anatomical models. These applications demonstrate VR/AR’s potential to improve patient outcomes and healthcare delivery efficiency. The prototype’s user interface considerations around comfort, clarity, and accessibility mirror healthcare applications’ emphasis on patient safety and clinical efficacy. The prototype’s design process documentation serves pedagogical purposes, making explicit the connection between system architecture, design decisions, and application domain requirements. By understanding how design choices support specific use cases, students and evaluators can assess the prototype’s suitability for different deployment contexts and identify modifications necessary for specialized applications.
